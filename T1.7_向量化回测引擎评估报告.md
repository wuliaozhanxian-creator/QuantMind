# T1.7 向量化回测引擎评估与一致性验证报告

> 汇报人：子agent-3（量化策略工程师）
> 汇报对象：项目经理 agent2号
> 评估依据：纯静态代码分析（未运行回测，本地无 Qlib 环境）
> 评估范围：`backend/shared/vectorized_backtest/engine.py` vs `backend/services/engine/qlib_app/services/backtest_service_runtime.py` + `utils/cn_exchange.py`
> 日期：2026-07-04

---

## 0. 结论速览（TL;DR）

- **是否建议重新启用？** → **有条件是**。现状下**不能**作为 Qlib 的等价替代直接启用；仅可作为"粗筛/预筛"加速路径，针对**窄场景（TopK 等权、纯多头、日线）**重新启用，且必须满足三个前置条件：输出契约适配、费用模型对齐、A/B 一致性测试补齐。
- **十年 ≤ 30s 目标可达性**：向量化引擎本身可在 **2~8s** 内完成单标的/单组合十年回测（纯矩阵运算），达成 30s 目标绰绰有余；但**仅限窄场景**，且结果与 Qlib 存在系统性偏差（非"一致"，而是"近似"）。Qlib 步进回测十年通常在分钟级，30s 目标**无法**通过 Qlib 原生达成。
- **关键风险**：向量化引擎当前 `portfolio_dict` 是 `"dummy"`，**完全不兼容** `RiskAnalyzer` 契约；费用模型缺印花税/最低佣金/过户费/涨跌停/整手取整；仅支持 1 种策略（Qlib 支持 10+ 种）。
- **工时估算**：完整适配（含一致性测试）约 **13~21 人日（3~4 周）**。

## 1. 两套引擎对比表

### 1.1 接口与实现对比

| 维度 | VectorizedBacktestEngine | Qlib 步进回测（SimulatorExecutor + CnExchange） |
|---|---|---|
| **源文件** | `backend/shared/vectorized_backtest/engine.py`（123 行） | `backtest_service_runtime.py` L499-514 + `utils/cn_exchange.py`（423 行） |
| **实现技术** | 纯 pandas/numpy 矩阵运算（unstack/rank/dot/cumprod） | Qlib `SimulatorExecutor`（`time_per_step="day"`）逐日 Python 循环 + 自定义 `CnExchange` 撮合 |
| **输入：信号** | `signals: DataFrame` MultiIndex(datetime, instrument)[score] | `signal_data`（同结构）→ 注入 strategy kwargs → `create_signal_from` |
| **输入：价格** | `prices: DataFrame` MultiIndex(datetime, instrument)[$close] | Qlib `D.features`（$open/$close/$volume/$factor） |
| **输入：配置** | `VectorizedBacktestConfig`（initial_capital, commission, slippage, topk） | `QlibBacktestRequest`（30+ 字段：deal_price, commission, min_commission, stamp_duty, transfer_fee, impact_cost_coefficient, signal_lag_days, rebalance_days...） |
| **输出结构** | `VectorizedBacktestResult`（annual_return/sharpe/max_drawdown/total_return/win_rate + `portfolio_dict={"dummy": df}`） | Qlib `backtest()` → `portfolio_dict` 含 `1day`→(report, indicator)、`report`、`trades`、`positions`、`final_value`、`account`、`position_value`、`sys_analyser` |
| **输出契约兼容性** | **不兼容** RiskAnalyzer（dummy 结构无法提取 report/trades/positions） | **原生兼容** RiskAnalyzer |
| **核心算法** | 信号→截面 rank→TopK 等权→权重×次日收益率→减换手成本→cumprod | 逐日：strategy.generate_trade_decisions → executor.execute → CnExchange.deal_order → 持仓/账户更新 |
| **支持策略类型** | **仅 1 种**：TopK 等权（截面 rank，`ranks <= topk`） | **10+ 种**：TopkDropout、WeightStrategy、VolatilityWeighted、long_short_topk、value_growth、alpha_cross_section、score_weighted、StopLoss、RiskGuardTopk、DeepTimeSeries、AdaptiveDrift、CustomStrategy |
| **做空/融券** | 不支持（纯多头等权） | 支持（`allow_short_selling`、margin Position、long/short exposure、融资融券费率） |
| **性能特征** | 十年×5000标的 ≈ **2~8s**（单次矩阵 pass） | 十年≈2400+ 交易日 × Python 逐日循环 ≈ **分钟级**（瓶颈所在） |

### 1.2 费用与撮合模型对比

| 维度 | VectorizedBacktestEngine | Qlib + CnExchange |
|---|---|---|
| **成交价** | `$close` 收盘价，`pct_change().shift(-1)`（次日收盘-收盘收益率） | 可配 `open`/`close`；无效时回退 close；含 10 日 lookback 回退；含复权因子 `$factor` 调整 |
| **前视偏差纠正** | 无（用当日 close 信号 + 次日收益，隐式 T+1） | `signal_lag_days=1`（T 日信号 T+1 生效）+ `deal_price` 可选 open |
| **佣金** | `turnover × commission`（扁平费率，默认 0.001） | `max(trade_val × commission, min_commission=5)`（费率+最低 5 元） |
| **印花税** | **无** | 卖出单边 0.05% |
| **过户费** | **无** | SH 单边，`max(val×rate, min_transfer_fee=0.01)` |
| **滑点/冲击成本** | `turnover × slippage`（扁平，默认 0.0001） | 平方根定律：`val × impact_coeff × sqrt(参与率)`，基于当日成交量 |
| **涨跌停** | **无** | `limit_threshold=0.095`，涨跌停跳过交易 |
| **整手取整** | **无**（分数权重） | 100 股/手整百取整（含复权因子处理） |
| **现金约束** | **无**（权重恒归一，无现金上限） | 买入受现金限制，迭代计算最大可买量 |
| **成交量裁剪** | **无** | `_clip_amount_by_volume` |
| **T+1 结算** | 隐式（shift(-1)） | 显式（day-step + rebalance_days，默认 3） |
| **benchmark** | 无 | 主回测禁用，分析阶段独立计算 |

## 2. 一致性风险点（按严重程度排序）

> 结论先行：两套引擎**不存在"一致"关系**，最多达到"方向性近似"。以下任一风险都会导致年化/夏普/回撤出现**系统性偏差**，而非随机误差。

### 2.1 高危风险（会导致结果质变，不可忽略）

| # | 风险点 | 向量化 | Qlib | 偏差方向 |
|---|---|---|---|---|
| H1 | **输出契约不兼容** | `portfolio_dict={"dummy": df}` | Qlib report/trades/positions 完整结构 | 向量化结果**无法进入** RiskAnalyzer，所有下游指标（alpha/beta/IR/交易统计/分层归因）会丢失或为 0 |
| H2 | **成交价模型** | close-to-close 收益率 | 可选 open 成交 + 复权因子 + 回退 | close 成交存在前视偏差，会**系统性高估**收益（schema 注释已警告） |
| H3 | **印花税缺失** | 无 | 卖出 0.05% | 十年累积换手下，向量化**高估**收益（卖出越多偏差越大） |
| H4 | **最低佣金缺失** | 无 min_commission | max(rate, 5元) | 小单交易向量化**低估**成本，对低价股/小仓位偏差大 |
| H5 | **涨跌停未处理** | 无 | 9.5% 跳过 | 向量化会在涨跌停日"成交"不可能成交的单，**高估**收益 |
| H6 | **现金约束缺失** | 权重恒归一 | 现金限制 + 最大可买量迭代 | 向量化可"超买"，资金管理语义不同 |
| H7 | **策略覆盖面** | 仅 TopK 等权 | 10+ 种（含止损/风控/行业限制/做空/深度时序） | 向量化**无法替代** 90% 的策略类型 |
| H8 | **rebalance 频率** | 隐式每日 | 可配（默认 3 天）+ n_drop | 调仓频率不同 → 换手率/成本/收益全部不同 |

### 2.2 中危风险（会累积偏差，长周期下显著）

| # | 风险点 | 偏差说明 |
|---|---|---|
| M1 | **整手取整缺失** | 向量化用分数权重，Qlib 按 100 股取整；十年累积产生持仓漂移 |
| M2 | **过户费缺失** | SH 标的无过户费，对纯 SH 池有累积偏差 |
| M3 | **冲击成本模型差异** | 扁平 slippage vs 平方根定律（基于成交量参与率）；大资金下 Qlib 冲击成本更高 |
| M4 | **T+1 语义差异** | 向量化 shift(-1) 是"次日收益"，Qlib 是"次日成交价"；对 TopK 等权近似等价，但对带 n_drop 的 TopkDropout 不等价 |
| M5 | **复权因子缺失** | 向量化直接用 `$close` 不做复权调整，Qlib 通过 `get_factor` 调整；有除权除息的标的会产生跳空 |

### 2.3 低危风险（可对齐或影响小）

| # | 风险点 | 说明 |
|---|---|---|
| L1 | benchmark 处理 | 两者均在分析阶段独立计算，可对齐 |
| L2 | 无风险利率 | 向量化硬编码 0.02，Qlib 走 request.risk_free_rate（默认 0.02），一致 |
| L3 | Sharpe 年化 | 两者均用 `daily_std × sqrt(252)`，一致 |

### 2.4 一致性测试现状

**无任何 A/B 对比测试**。`backend/tests/` 下不存在 `test_vectorized_backtest` 或任何对比用例。`engine_manager.compare_engines` 是设计中的对比入口，但因属性缺失/拼写错误**无法运行**。这意味着即便重新启用，也**没有自动化机制**验证两套引擎结果一致性。

## 3. 重新启用建议

### 3.1 总体结论：**有条件是**

**不建议**作为 Qlib 的等价替代"直接重新启用"；**建议**作为**窄场景粗筛加速路径**重新启用，前提是满足下列前置条件，并明确"粗筛/精确"两层定位。

### 3.2 适用场景（仅限以下场景可启用）

- 策略类型 = **TopkDropout 退化为 TopK 等权**（`n_drop=topk`，纯多头）
- `deal_price` 接受 close（或可接受 close 近似）
- 纯多头、无做空、无止损、无行业限制
- 用于**参数网格搜索/遗传算法的初筛阶段**（`QlibOptimizationRequest`/`QlibGeneticOptimizationRequest` 的子任务，已有 `history_source="optimization"` 标记可复用）
- 用户**显式接受**精度折损（API 层加 disclaimer + feature flag）

### 3.3 必须完成的适配工作

| 工作项 | 内容 | 工时（人日） |
|---|---|---|
| **A. 输出契约适配器** | 将 `VectorizedBacktestResult` 转换为 Qlib `portfolio_dict` 结构（`1day`→(report, indicator)、report 含 account/return/turnover/total_turnover 列、trades、positions、final_value），使 RiskAnalyzer 可直接消费 | 2~3 |
| **B. 费用模型对齐** | 在向量化路径补齐：min_commission、stamp_duty（卖单）、transfer_fee（SH）、impact_cost（近似）；将扁平 slippage 升级为参与率模型或显式标注为简化模型 | 2~3 |
| **C. 成交价与复权对齐** | 支持 `deal_price=open`（用 $open 计算次日成交）、引入 `$factor` 复权、补涨跌停过滤（用 $close 涨跌幅 >9.5% 置零换手） | 2~3 |
| **D. T+1/rebalance 对齐** | 按 `rebalance_days` 重采样权重（非每日）、按 `n_drop` 实现 TopkDropout 近似、`signal_lag_days` 对齐 | 1~2 |
| **E. 现金/整手约束** | 整百取整、现金上限裁剪（向量化近似：按权重×资金反推股数后取整） | 2~3 |
| **F. A/B 一致性测试套件（关键）** | 构造固定信号+价格的 golden case，对同一 TopK 等权场景跑双引擎，断言 annual_return/sharpe/max_drawdown 相对偏差 < 阈值（建议 5%~10%）；覆盖 1年/5年/10年三档 | 3~5 |
| **G. Feature Flag + API 集成** | `use_vectorized` 真正生效（替换 L499-507 的强制重定向），加 `vectorized_mode="coarse"/"precise"`，加 disclaimer，限制策略类型白名单 | 1~2 |
| **合计** | | **13~21 人日（约 3~4 周）** |

### 3.4 预估启用后单标的十年回测耗时

| 场景 | 预估耗时 | 说明 |
|---|---|---|
| 纯矩阵 pass（现状） | **2~8s** | unstack+rank+dot+cumprod，十年×5000标的 |
| 补齐费用/涨跌停/取整后 | **5~15s** | 增加若干向量化掩码与条件运算，仍远低于 30s |
| 含输出契约转换 | **+1~3s** | 构造 Qlib report DataFrame |
| **综合** | **≈ 8~18s** | **达成 ≤30s 目标** |

> 注：以上为静态推断。向量化引擎瓶颈在 `unstack`/`rank`/`dot` 三处矩阵运算，复杂度 O(N_days × N_inst)，与交易日数线性相关但常数极小。十年 2400+ 交易日对矩阵运算无压力。

### 3.5 重新启用路线图建议

1. **Phase 1（1 周）**：先做 F（一致性测试）的最小版本 + A（输出适配器），用现状引擎跑通"能产出可比指标"的闭环，量化当前偏差幅度（这一步是决策依据，**必须先做**）。
2. **决策点**：若 Phase 1 测得 TopK 等权场景下年化相对偏差 < 10%，则继续 Phase 2；否则放弃重新启用，转替代方案。
3. **Phase 2（2~3 周）**：B+C+D+E 逐项对齐 + 扩展一致性测试覆盖。
4. **Phase 3（3~5 天）**：G 集成 + 灰度（仅 optimization 子任务启用）+ 监控。

## 4. 若不建议全面启用：替代方案

鉴于"有条件是"的窄场景限制，若委托方追求**全策略类型**的 30s 目标，向量化引擎无法覆盖，需走以下替代路径（可与向量化粗筛并行）：

### 4.1 Qlib 自身优化（推荐主线）

| 方向 | 说明 | 预期收益 |
|---|---|---|
| **特征/信号缓存** | `_build_signal_data` 反复从 pred.pkl/特征库读取，缓存 signal_data + D.features 结果到 Redis（DB=4 backtest） | 减少 I/O，单次省 5~15s |
| **Numpy 后端 / 矩阵化 Executor** | 评估 `qlib.contrib` 的 numpy 风格 executor，或自研矩阵化 SimulatorExecutor 子类（保留 CnExchange 费用语义，但批量化日内循环） | 十年或降至 30~60s |
| **裁剪不必要的指标计算** | `generate_portfolio_metrics=True` + RiskAnalyzer 的分层归因/风格归因耗时大；粗筛阶段关闭高级统计 | 省 20%~40% |
| **并行参数搜索** | `QlibOptimizationRequest.max_parallel` 已支持，配合进程池压满 CPU | 网格搜索总耗时下降 |

### 4.2 两层回测架构（推荐落地形态）

```
用户请求
   │
   ├─ 参数优化/初筛（history_source=optimization）
   │     └─ 向量化粗筛引擎（feature flag，TopK 等权白名单）→ 秒级
   │           └─ 取 Top-N 参数组合
   │
   └─ 最终精确回测（history_source=manual）
         └─ Qlib 步进回测（CnExchange 全费用）→ 分钟级，作为权威结果
```

- 粗筛层负责"快"，精确层负责"准"。
- 两层结果均持久化，前端标注"粗筛/精确"来源。
- 这与 M1 评审"决策四"（允许评估一致性后作为可选加速路径）的精神一致：**加速路径只承担筛选，不承担出报告**。

### 4.3 不推荐的路径

- **直接删除向量化引擎**：虽为死代码，但其矩阵化思路是未来 Numpy Executor 的参考实现，保留有知识价值；建议保留但加 `@deprecated` 注释与"仅粗筛"说明。
- **全量改写向量化引擎以支持 10+ 策略**：工作量远超 Qlib 优化，且止损/风控/做空等逻辑本质是事件驱动的，强行向量化会引入更多偏差。

## 5. 子agent-3 专业建议（向项目经理汇报要点）

1. **30s 目标可达性结论**：
   - 向量化引擎**能**达成 ≤30s（预估 8~18s），但**仅限 TopK 等权纯多头窄场景**，且结果是"近似"而非"一致"。
   - Qlib 步进回测**不能**达成 ≤30s（分钟级），是十年回测的权威路径。
   - **建议向委托方汇报**：30s 目标在"粗筛/精确两层架构"下可达；若要求全策略类型 30s 精确回测，则**不可达**，需放宽为"粗筛 ≤30s + 精确 ≤5min"。

2. **重新启用决策**：**有条件是**。前置条件 = 输出契约适配 + 费用对齐 + A/B 一致性测试 + feature flag + 策略白名单。工时 13~21 人日。**强烈建议先做 Phase 1（一致性测试最小版）以数据驱动决策**，避免投入后才发现偏差过大。

3. **最大风险**：当前 `engine_manager._run_vectorized_backtest` 与 `compare_engines` 是**已损坏的死代码**（属性不存在 + 拼写错误），若有人误以为可用而直接调用会崩溃。建议在重新启用前**先修复或显式标注禁用**，避免误用。

4. **与 M1 决策四的契合**：本评估印证了"评估一致性后作为可选加速路径"的审慎态度——一致性**不成立**（只能近似），因此只能作为"可选加速路径"而非"替代路径"，与决策四措辞一致。

5. **股票代码规范**：本评估涉及的 `CnExchange` 已使用 SH 前缀判断过户费（`stock_id.upper().startswith("SH")`），与项目 SH 前缀规范一致；向量化引擎不涉及代码格式逻辑，无规范风险。
