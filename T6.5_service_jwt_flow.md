## 一、背景与现状

仓库当前服务间认证采用 **共享密钥 + X-Internal-Call header** 方案（T6.2 已移除其后门绕过逻辑，
仅保留对 `/api/v1/internal/*` 专用路径的校验）：

- 调用方在请求头附加 `X-Internal-Call: <INTERNAL_CALL_SECRET>`
- 被调用方通过 `backend/shared/auth.py::get_internal_call_secret()` 读取相同密钥并比对

### 1.1 现状的安全缺陷

| 缺陷 | 说明 |
|------|------|
| 无调用方身份 | 所有服务共用同一密钥，被调用方无法区分请求来自 api / engine / trade |
| 无时效 | 密钥不过期，泄漏后窗口期长，轮换需所有服务同步重启 |
| 密钥频传 | 每次请求都明文传输共享密钥，日志/中间件泄漏面大 |
| 无审计 | 无法追溯某次内部调用由哪个服务发起 |
| 轮换困难 | 改密钥需全集群协同，运维成本高 |

### 1.2 T6.2 已铺垫的过渡件

`backend/shared/auth.py` 中已存在两个过渡函数（T6.2 引入）：

- `create_service_token(service_name, expire_seconds=300) -> str`
  - 用 `SECRET_KEY`（环境变量）签发短期 JWT，payload 含 `service` / `iat` / `exp`
  - **当前状态：已实现但尚未被任何调用方使用**（grep 确认仅有定义，无调用点）
- `decode_jwt_token(token) -> dict`
  - 用 `SECRET_KEY` 验证签名与过期，返回 payload
  - **当前状态：已被 6 处使用**（request_logging、trade/deps、stream/ws_core、engine/qlib_app/websocket、engine/qlib_app/risk_monitor、api/community/auth）

## 三、目标 Service JWT 认证流程

### 3.1 总体流程图

```
┌──────────────────────────────────────────────────────────────────────┐
│                        Service JWT 认证流程                          │
└──────────────────────────────────────────────────────────────────────┘

  调用方服务 (api / engine / trade)                  被调用方服务
 ┌───────────────────────────────┐               ┌──────────────────────────┐
 │ 1. 发起服务间调用前            │               │ 4. 收到请求              │
 │    token = create_service_token( │   ───────▶   │    抽取 Authorization:   │
 │        service_name="api",     │   HTTP 请求    │    Bearer <service_jwt>  │
 │        expire_seconds=300)     │   携带 header: │                          │
 │                               │   Authorization│ 5. verify_service_token( │
 │ 2. 得到短期 JWT                │   : Bearer     │      token,             │
 │    payload={service:"api",     │   <service_jwt>│      allowed_services=  │
 │              iat, exp(+5min)}  │               │      ["api","engine"])   │
 │                               │               │                          │
 │ 3. 注入请求头                  │               │ 6. 内部调用 decode_jwt_   │
 │    Authorization: Bearer ...   │               │    token() 验签+过期     │
 │    （X-User-Id / X-Tenant-Id   │               │    校验 service 声明 ∈    │
 │     透传用户上下文）            │               │    allowed_services      │
 └───────────────────────────────┘               └──────────────────────────┘
                                                          │
                                                          ▼
                                          ┌───────────────────────────────┐
                                          │ 7a. 通过 → 注入 request.state │
                                          │     service="api"，放行       │
                                          │ 7b. 失败 → 401 Unauthorized   │
                                          │     （验签失败/过期/服务不在   │
                                          │      允许列表）                │
                                          └───────────────────────────────┘
```

### 3.2 签发流程（调用方）

```
create_service_token("api", expire_seconds=300)
        │
        ▼
  读取 SECRET_KEY (env: SECRET_KEY/JWT_SECRET_KEY/JWT_SECRET)
        │  未配置 → raise RuntimeError (fail-fast，T6.2 风格)
        ▼
  读取 ALGORITHM (env: ALGORITHM/JWT_ALGORITHM，默认 HS256)
        │
        ▼
  payload = {
      "service": "api",          # 调用方身份
      "iat": now,                # 签发时间
      "exp": now + 300s          # 过期时间（默认 5 分钟）
  }
        │
        ▼
  jose_jwt.encode(payload, secret_key, algorithm) → JWT 字符串
```

### 3.3 验证流程（被调用方）

```
收到 Authorization: Bearer <token>
        │
        ▼
  verify_service_token(token, allowed_services=["api","engine"])
        │
        ▼
  payload = decode_jwt_token(token)
        │  ├─ SECRET_KEY 未配置 → RuntimeError (fail-fast)
        │  ├─ 签名无效 / 已过期 → HTTPException 401
        │  └─ 成功 → 返回 payload
        ▼
  service = payload.get("service")
        │
        ▼
  service ∈ allowed_services ?
        ├─ 是 → 返回 payload（认证通过）
        └─ 否 → HTTPException 403（调用方无权访问该端点）
```

### 3.4 过期与续签处理

- **短期令牌**：默认 5 分钟过期，调用方每次发起调用前重新签发（无需持久化存储 service token）
- **无刷新令牌**：service JWT 是无状态的短期凭证，不设 refresh token，过期直接重新签发
- **密钥轮换**：所有服务读取相同 `SECRET_KEY` 环境变量，轮换时同步更新 `.env` 并重启服务即可；因 token 短期，轮换后 5 分钟内全部自然失效
- **紧急撤销**：若需立即失效，轮换 `SECRET_KEY`（所有在途 service JWT 立即验签失败）

## 五、迁移路径（X-Internal-Call → Service JWT）

> 遵循指令②：逐路由迁移，每路由单独 PR 便于回滚。
> 遵循指令⑤：迁移完成前不删除 shared/auth.py 与 get_internal_call_secret。

### 5.1 第一阶段：补齐基础设施（1 个 PR）

在 `backend/shared/auth.py` 新增两个函数（不删除任何现有代码）：

```python
def verify_service_token(token: str, allowed_services: list[str]) -> dict:
    """验证 service JWT 并校验调用方服务是否在允许列表内。"""
    payload = decode_jwt_token(token)          # 复用 T6.2 验签
    service = payload.get("service")
    if service not in allowed_services:
        raise HTTPException(
            status_code=status.HTTP_403_FORBIDDEN,
            detail=f"Service '{service}' not allowed for this endpoint",
        )
    return payload


def require_service_token(*allowed_services: str):
    """FastAPI 依赖：要求请求携带合法 service JWT 且调用方在允许列表内。"""
    def checker(
        credentials: HTTPAuthorizationCredentials | None = Depends(
            HTTPBearer(auto_error=False)
        ),
    ) -> dict:
        if not credentials:
            raise HTTPException(401, "Missing service token")
        return verify_service_token(credentials.credentials, list(allowed_services))
    return checker
```

### 5.2 第二阶段：并行运行（每个服务 1 个 PR）

被调用方端点同时接受两种凭证（兼容期）：

```python
# 被调用方内部端点（兼容期）
def _auth_internal(request):
    # 优先 service JWT
    auth = request.headers.get("Authorization", "")
    if auth.startswith("Bearer "):
        try:
            return verify_service_token(auth[7:], ["api", "engine", "trade"])
        except HTTPException:
            pass
    # 回退 X-Internal-Call（过渡）
    if request.headers.get("X-Internal-Call") == get_internal_call_secret():
        return {"service": "legacy"}
    raise HTTPException(401, "Invalid internal credentials")
```

调用方逐步切换为 service JWT：

```python
# 调用方
headers = {"Authorization": f"Bearer {create_service_token('api')}"}
```

### 5.3 第三阶段：逐路由 cutover（每路由 1 个 PR）

按 2.1 清单逐路由：
1. 调用方改为只发 service JWT（移除 X-Internal-Call）

## 六、安全收益对比

| 维度 | X-Internal-Call（现状） | Service JWT（目标） |
|------|------------------------|---------------------|
| 调用方身份 | 无（共享密钥） | 有（`service` 声明） |
| 时效 | 永久 | 5 分钟过期 |
| 密钥传输 | 每次明文传密钥 | 仅传签名令牌 |
| 端点级授权 | 无法按调用方限制 | `allowed_services` 白名单 |
| 审计 | 无法溯源 | payload 可记录调用方 |
| 轮换 | 全集群同步重启 | 改 SECRET_KEY，5 分钟自然失效 |
| 撤销 | 无法单点撤销 | 轮换密钥即全部失效 |

## 七、遗留决策项（需项目经理 agent2号 评审）

1. **header 命名**：service JWT 是否复用标准 `Authorization: Bearer`，还是用专用 `X-Service-Token`？
   - 复用 Authorization：与用户 JWT 同 header，被调用方需区分 token 类型（看 `service` vs `sub` 声明）
   - 专用 header：语义清晰，但偏离标准。**建议专用 `X-Service-Token`** 避免与用户 JWT 混淆。
2. **k8s_manager 环境注入**：Pod 内服务如何获取 service JWT 签发能力？建议 Pod 共享 SECRET_KEY 即可自行签发。
3. **`optional_auth` 死代码**：本轮未删（不在指令明确范围），建议第四阶段一并清理。
4. **`create_service_token` 当前未被调用**：作为过渡件已就绪，待第二阶段开始接入。
