## 3. 根因分析

### 3.1 结论：设计差异（非 Bug）

偏差根因是 **VectorizedBacktestEngine 与 qlib SimulatorExecutor 在涨跌停权重处理上的两层设计差异**，不是代码 Bug。

### 3.2 根因1：权重归一化差异（主要因素，约占 45-78% 偏差）

**VectorizedBacktestEngine 的 `_sequential_limit_carry` 方法**：

```python
# engine.py 第 195-210 行
carried = tgt.where(~nt, prev_w)   # 非可交易 -> 沿用前日归一化权重
s = float(carried.sum())
w = carried / s if s > 0 else carried  # 重新归一化，权重和 = 1.0
```

- 遇到涨跌停时，对（携带后的）全部目标权重**重新归一化**，使权重和恒等于 1.0
- 涨停（新买拒绝）：目标权重置 0，剩余股票权重被**放大**（重新归一化后 > 1/topk）
- 跌停（卖出拒绝）：沿用前日归一化权重，剩余股票权重被**调整**

**qlib SimulatorExecutor**：

- 涨跌停时仅拒绝该笔订单，**不重新归一化**其他股票的权重
- 拒绝买入 -> 资金以现金形式留存（总权重 < 1.0）
- 拒绝卖出 -> 仓位锁定在当前持仓量（按实际股价估值）

**归一化影响量化**（实验4数据）：

| topk | vec vs no_norm median 偏差 | 实际 vec vs qlib 偏差 | 归一化贡献占比 |
|------|--------------------------|---------------------|--------------|
| 5 | 10.73% | 13.67% | 78% |
| 10 | 9.50% | 21.04% | 45% |

### 3.3 根因2：持仓估值模型差异（次要因素，约占 22-55% 偏差）

**VectorizedBacktestEngine**：基于**目标权重**计算收益
```python
portfolio_daily_returns = (weights * asset_returns).sum(axis=1)
```
- 权重是每日重新设定的目标权重（1/topk 或归一化后的值）
- 不考虑持仓市值随股价变化产生的漂移

**qlib SimulatorExecutor**：基于**实际持仓**计算收益
- 持仓以股数 x 当前股价估值，权重随股价变化自然漂移
- 每日调仓时，executor 先卖后买，按可用现金分配
- 当涨跌停导致卖不出/买不进时，实际持仓偏离目标权重

**此差异在 topk=10 时被放大的原因**：

| 指标 | topk=5 | topk=10 | 倍数 |
|------|--------|---------|------|
| 涨跌停在 top-k 中的天数 | 15-19 / 197 | 24-35 / 197 | ~1.8x |
| 持仓股票数 | 5/15 = 33% | 10/15 = 67% | 2.0x |
| 归一化贡献占比 | 78% | 45% | -- |
| 持仓估值贡献占比 | 22% | 55% | 2.5x |

topk=10 时持仓占比 67%（10/15 只股票），远高于 topk=5 的 33%。更多的持仓股票意味着：
1. 更高的涨跌停概率（更多暴露）
2. 更多的持仓市值漂移（更多股票价格变化）
3. 更多的调仓摩擦（更多买卖订单受涨跌停影响）

### 3.4 排除的因素

以下因素经实验验证**不是**根因：

| 假设 | 实验 | 结论 |
|------|------|------|
| 选股差异（only_tradable=True 先过滤再选 top-k） | 实验3：only_tradable=True vs False | **排除**。两种模式产生完全相同的结果（0% 差异），因为 qlib executor 在执行层也会拒绝非可交易股票的订单 |
| 信号差异（topk=10 排序边界更敏感） | 实验2：逐日选股对比 | **排除**。信号生成逻辑完全一致，均使用 `rank(method="first")` |
| 数值精度（1/10 vs 1/5 权重精度） | 实验4：权重差异分析 | **排除**。avg_weight_diff 仅 0.013，且 topk=5/10 差异不大 |
| 交易日对齐 | 代码审查 | **排除**。两个引擎使用同一份数据，交易日历完全一致 |
| 交易成本差异 | 代码审查 | **排除**。两引擎均使用 0.0004 单边费率，计算方式等价 |

## 5. 是否有 Bug

**无 Bug。** 这是向量化引擎与事件驱动引擎之间的设计差异。

### 设计差异说明

| 维度 | VectorizedBacktestEngine | qlib SimulatorExecutor |
|------|------------------------|----------------------|
| 权重模型 | 目标权重（每日重设） | 实际持仓（股数 x 股价） |
| 涨跌停处理 | 归一化（权重和=1.0） | 拒绝订单（可能留现金） |
| 调仓执行 | 向量化矩阵运算 | 事件驱动先卖后买 |
| 现金管理 | 始终 100% 投资 | 可能有现金残留 |

`_sequential_limit_carry` 的代码注释声称"忠实 Qlib CnExchange 行为"，但归一化步骤是一个简化近似，并不完全匹配 qlib executor 的实际行为（executor 不归一化）。

## 6. 修复方案/建议

### 方案 A：维持现状（推荐）

- 结论：21.04% 偏差是设计差异的预期结果，非 Bug
- 当前 topk=5 主验证方案（15% 阈值）已覆盖核心场景
- topk=10 偏差根因明确，无需修复

### 方案 B：调整 topk=10 阈值（如需 topk=10 验证）

- 将 topk=10 的均值回归偏差阈值从 15% 调整为 **25%**
- 依据：归一化贡献 ~10% + 持仓估值贡献 ~11% = ~21% 预期偏差

### 方案 C：修改 _sequential_limit_carry 去掉归一化（不推荐）

- 将 `w = carried / s` 改为 `w = carried`（不归一化）
- 预期效果：topk=10 median 偏差从 21.04% 降至 ~14.5%
- 风险：可能影响其他场景（趋势跟踪 topk=5/10、多因子）的已有结果
- 违反约束："不修改 VectorizedBacktestEngine 的核心逻辑（除非确认有 Bug）"
